## Description
`BuslaneLinker` currently hand-maintains separate source-to-target maps for
Buslane type constructors, type parameters, values, data constructors, effects,
and operations inside `modules/lanec/compile/module_compile.mbt`. Remapping a
linked program requires maintainers to understand metadata allocation order,
effect-to-operation ownership, import seeding, exports, fingerprints, terms,
types, effects, and expressions in one shallow module.

This shape already caused a real bug: remapping an operation can recursively
remap its owner effect, and effect remapping initializes the effect's operations.
Without a second lookup after owner remap, the same source operation was
allocated twice and handler dispatch saw mismatched operation identities.

## Design
Introduce a deeper remapping seam for linked Buslane identities. The seam should
centralize reserve/lookup/fill behavior for all Buslane identity kinds and make
reentrant owner initialization safe by construction. It should hide the
operation/effect and type/data-constructor ownership coupling from callers such
as expression remapping.

Non-goals:
- Do not change the public module interface/object artifact format unless the
  remapper design requires a separately reviewed schema change.
- Do not merge Buslane metadata into source-level symbols; the linked program
  must still contain Buslane identities.
