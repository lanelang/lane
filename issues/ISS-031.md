## Description
Buslane needs first-class effect metadata before `perform`, handlers, verifier
rules, or interpreter behavior can be implemented. Add the core identity and
type-level representation needed by the effect roadmap.

## Design
Introduce `EffectId` and `OperationId`, effect constructor metadata, operation
signature metadata, `Kind::Effect`, and effect terms for empty effects,
singleton effects, effect row variables, and unions. Reuse `TypeParameterId`
with `Kind::Effect` for effect row variables; do not introduce a separate
runtime identity family.

## Acceptance Criteria
- [ ] Buslane metadata can represent effect constructors and operation
  signatures.
- [ ] Buslane kinds include `Effect`.
- [ ] Buslane type-level effect terms represent empty, singleton, row-variable,
  and union forms.
- [ ] Effect row variables use `TypeParameterId` with `Kind::Effect`.
- [ ] `moon info && moon fmt` has been run after the change.
- [ ] Relevant Buslane model tests or snapshots cover the new metadata.
