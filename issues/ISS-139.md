## Description
After `Double` exists as a primitive value, Basic should provide the ordinary arithmetic and comparison operations for `Double` through explicit `%f64_*` builtins and contextual operation offers.

## Design
Add Basic builtin bindings for `%f64_add`, `%f64_sub`, `%f64_mul`, `%f64_div`, `%f64_neg`, `%f64_equal`, and `%f64_less`. Extend `Basic.Ops` so `Double` participates in `Add`, `Sub`, `Mul`, `Div`, `Neg`, and comparison offers using IEEE floating-point equality and ordering. Do not add `Rem[Double]`.

Extend the Lane runtime/interpreter builtin implementation for the same `%f64_*` operation names so run examples can execute. Keep this batch independent from special constants: no `%f64_inf`, `%f64_ninf`, or `%f64_nan` surface here unless needed internally for tests.

Non-goals: `Double` special constants, total floating-point ordering, printing/conversion helpers, and implicit or explicit `Int`/`Double` conversions.

## Acceptance Criteria
- [x] Basic exposes `Double` arithmetic and comparison offers through `Basic.Builtins` and `Basic.Ops`.
- [x] Runtime/interpreter handles `%f64_add`, `%f64_sub`, `%f64_mul`, `%f64_div`, `%f64_neg`, `%f64_equal`, and `%f64_less`.
- [x] Valid run examples cover arithmetic and comparison, printing only fixed success strings.
- [x] Invalid examples show that `Double` remainder is not available.
- [x] Lane and `../basic` validation pass, including `tools/check-lane-run-examples.sh`.
