## Description
Implement `extern type T : Type` and `extern type E : Effect` declarations with distinct semantic flavors and update extern function validation to accept closed rows of External Effects and direct External Types.

## Design
Reuse ordinary declaration naming, visibility, import, export, qualification, and alias rules. External Types are zero-parameter nominal types; External Effects may use ordinary supported kind parameters. Preserve explicit flavor rather than inferring it from declaration shape. Expand transparent aliases before classifying direct host parameters and results.

## Acceptance Criteria
- [ ] Parser, formatter, resolver, and diagnostics support both declaration forms.
- [ ] External Type declarations reject parameters, bodies, constructors, and layouts.
- [ ] External Effects are non-handleable, non-monadic residual effects.
- [ ] Extern signatures accept primitives, zero-parameter External Types, and closed rows containing only External Effects.
- [ ] Ordinary abstract or aggregate types remain invalid at the host boundary.
- [ ] Focused valid and invalid source tests pass.
