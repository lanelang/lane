# ISS-124: Adopt typed diagnostic ownership across compiler phases

## Description
Lane already has a shared compiler diagnostic data model and adapter, but diagnostic wording is still largely centralized in `modules/lanec/compile/diagnostic.mbt`. That makes compile orchestration understand parser, resolve, typecheck, link, artifact, and lint wording, and it keeps growing whenever a phase adds a new diagnostic.

Move toward the rustc-style model: diagnostics remain structured and uniformly rendered, while each compiler phase owns typed diagnostic conversion near the semantic data that produced the diagnostic.

## Design
Use handwritten MoonBit trait implementations instead of derive macros. `modules/lanec/diagnostic` should define the shared builder/context/trait surface. Parser, resolve, typecheck, and compile/link/artifact diagnostics should implement conversion near their owner packages. Do not introduce a global string registry.

Non-goals:
- Do not change diagnostic rendering or LSP transport semantics.
- Do not centralize all messages into one string table.
- Do not redesign typecheck, resolve, or parser diagnostic kinds.

## Acceptance Criteria
- [ ] `lanec/diagnostic` exposes a small typed diagnostic conversion API usable by compiler phases.
- [ ] Parser diagnostics no longer require `compile/diagnostic.mbt` to know parser wording.
- [ ] Resolve diagnostics no longer require `compile/diagnostic.mbt` to know resolve wording.
- [ ] Typecheck diagnostics no longer require `compile/diagnostic.mbt` to know typecheck wording.
- [ ] Compile/link/artifact diagnostics remain owner-local and do not depend on source-language phase wording.
- [ ] Golden, LSP, and example-smoke diagnostics remain stable or improve intentionally.
- [ ] `compile/diagnostic.mbt` is reduced to orchestration glue or removed.
