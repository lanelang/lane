## Description
Buslane effect core is now part of the source-to-Buslane path, but ANF lowering
still aborts on Buslane `perform` and `handle` expressions. This is explicit in
`modules/lanec/anf/lower_buslane.mbt` and means any workflow that requires ANF
after effect lowering is blocked.

If ANF remains out of the current execution path, this is a known feature gap.
If ANF becomes part of compile, optimization, or code generation for effectful
programs, this becomes a hard compiler failure.

## Design
Decide and implement the ANF representation for Buslane `perform`, `handle`, and
explicit resume continuations. Preserve deep handler semantics: resumed
continuations run under the same handler, while effects performed directly by
handler arm bodies are not self-captured.

Non-goals:
- Do not change source effect syntax or type checking in this issue.
- Do not implement unrelated ANF optimizations.

## Acceptance Criteria
- [ ] Buslane ANF has explicit representation or lowering strategy for `perform`.
- [ ] Buslane ANF has explicit representation or lowering strategy for `handle`.
- [ ] Resume continuation semantics are covered by tests, including multi-shot resume.
- [ ] Effectful Lane examples can pass through the ANF lowering path if that path is enabled for them.
- [ ] `moon test modules/lanec/anf modules/buslane/interpreter --warn-list +73` passes.
- [ ] `moon info && moon fmt` has been run after the change.
