## Description
`modules/lanec/compile/analysis_*.mbt` is large and serves multiple audiences:
compiler analysis results, LSP hover/definition/inlay/completion data, semantic
indexes, and checked/resolved traversal. This is a stronger architecture smell
than the file prefix alone: editor-facing analysis is mixed into the compile
facade, making ownership and public API direction harder to reason about.

## Design
Design the boundary before moving files. Decide whether analysis should become a
`compile/analysis` package, a sibling `lanec/analysis` package, or remain under
`compile` with a narrower facade. The decision must clarify:

- which package owns `AnalysisResult` and editor-oriented entries;
- whether `compile` re-exports analysis APIs or callers depend directly on the analysis package;
- how diagnostics, partial analysis, and source identity flow across the boundary;
- which existing LSP callers should change.

Avoid a mechanical file split. If shared types cause cycles, extract the shared
model deliberately rather than duplicating it.

## Acceptance Criteria
- [x] A concrete package/API boundary for analysis is implemented or documented with a deferral rationale.
- [x] LSP callers use the intended analysis boundary, not incidental compile internals.
- [x] Existing hover, definition, inlay, completion, and diagnostic behavior remains covered by tests.
- [x] Validation passes with `moon check --warn-list +73`, `moon test --warn-list +73`, and relevant LSP tests.
