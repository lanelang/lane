## Description
Syntax and formatter failures currently render terse text such as `lexical error: invalid string escape \q`, `parse error: unexpected RBrace, expected expression`, or `parser produced no source file`. These messages identify the parser failure but do not show what token was rejected, what the grammar expected in user terms, or how to fix common mistakes.

Current message inventory:

- `lexical error: illegal character ...`
- `lexical error: unterminated block comment`
- `lexical error: unterminated string`
- `lexical error: invalid string escape ...`
- `lexical error: invalid hex escape ...`
- `lexical error: non-ascii string character`
- `parse error: unexpected ...`
- `parse error: unexpected end of input`
- `parser produced no source file`
- formatter wrapper output such as `<file>: not formatted`

## Design
Move syntax diagnostics onto the structured report model and share rendering between `lane check` and `lane fmt`. For expected tokens, collapse raw grammar symbols into useful source-language labels. Add help for common cases: close the string/comment, remove or escape the character, add the missing expression or delimiter, and run `lane fmt` for formatting-only failures.

## Acceptance Criteria
- [ ] Lexer errors include a primary label pointing at the bad character or unterminated construct.
- [ ] Parser errors include the unexpected token/end-of-file and a concise expected-syntax list.
- [ ] Common lexical errors include specific help text.
- [ ] `lane fmt --check` and parse failures use the same diagnostic style as `lane check`.
- [ ] Golden tests cover representative lexer, parser, and formatter failures.
