# ISS-132: Select executable entry at link time and add core occurrence analysis

- Type: epic

## Description
Lane currently allows `runobj` to choose an entry from a linked program. That keeps linked artifacts in a multi-entry shape and prevents whole-program optimization from treating the executable entry as the root before final artifact emission.

Change the executable artifact model so `link` accepts a set of `.lmo` files and one exported entry such as `Main.main`, validates that entry, runs linked-core optimization analysis from that root, and emits a single-entry `.lbp`. `runobj` should only accept a `.lbp` file and execute the entry already selected by link.

## Design
Hard-cut the runtime entry-selection model instead of preserving both modes.

The intended pipeline:
- Link reads `.lmo` inputs and resolves the requested entry through an exported module symbol.
- Link builds an internal linked core value with imported references connected and exactly one selected entry.
- Link validates that the selected entry is executable, including type shape and supported runtime effects.
- `modules/lanec/occurrence` analyzes the internal linked Buslane/core value and returns optimizer-local occurrence facts.
- Future optimization passes consume those facts before final executable artifact emission.
- The emitted `.lbp` stores a selected executable payload; it does not expose a runtime-selectable public entry catalog and may omit type metadata needed only during link-time validation.
- `runobj` loads and executes a `.lbp`; it does not parse or validate an entry argument.

The first occurrence analysis should be a core-level analysis over linked Buslane/core value identities, not source unused analysis and not ANF liveness. It should record structured facts useful for inlining, DCE, and closure conversion: use count, call-position use, non-call escape, effectful-context use, and reachability from the selected entry.

Non-goals:
- Do not add final inlining, DCE, closure conversion, or bytecode optimization in this issue unless needed as a smoke consumer.
- Do not persist occurrence facts as `.lbp` semantic payload.
- Do not let `runobj` depend on source-level type metadata to decide whether an entry is executable.
- Do not allow private lowered definitions or Buslane implementation names as CLI entry contracts.

## Acceptance Criteria
- [ ] `lane link` accepts `.lmo` inputs and a required entry such as `Main.main`.
- [ ] Link resolves the entry through exported module symbols and rejects missing, non-public, or ambiguous entries with clear diagnostics.
- [ ] Link validates executable entry type and supported runtime effects before emitting `.lbp`.
- [ ] The link pipeline has an explicit internal linked-core or optimization-core value before artifact emission.
- [ ] `.lbp` stores exactly one selected executable entry/payload and no longer exposes a runtime-selectable entries array.
- [ ] `lane runobj` accepts only a `.lbp` file and no entry argument.
- [ ] A new `modules/lanec/occurrence` package exposes an analysis API over the internal linked core and returns structured occurrence facts.
- [ ] Occurrence analysis roots are the selected entry plus required runtime or external roots, not all public entries.
- [ ] Occurrence facts remain optimizer-local and are not serialized as artifact semantic payload.
- [ ] Regression tests cover old `runobj file:Entry` syntax rejection, missing link entry, invalid entry symbol, non-public entry, invalid executable type/effects, and a small occurrence summary.
- [ ] `moon info && moon fmt`, `moon check --warn-list +73`, `moon test --warn-list +73`, and `tools/check-lane-run-examples.sh` pass.
