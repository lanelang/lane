## Description
Allow top-level type aliases to infer Effect kind and transparently expand in function effect annotations and effect set literals.

## Design
Reuse the transparent alias model. Effect set aliases expand during typechecking, participate in alias cycle detection, support Effect-kind parameters with existing `Task[E]` application syntax, and normalize through the existing effect set normalization rules. Checked and lowered IR should contain ordinary effect sets, not alias identities.

## Acceptance Criteria
- [ ] `type UI = { Console, Log }` can be used as `! UI`.
- [ ] `type Task[E : Effect] = { Coroutine, E }` can be applied in effect annotations.
- [ ] Alias expansion detects cycles and reports source-oriented diagnostics.
- [ ] `Task[Int]` or Type-kind aliases in effect position produce kind mismatch diagnostics.
- [ ] Expanded effect sets are flattened, deduplicated, and order-insensitive according to existing effect equality.
