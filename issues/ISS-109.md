# ISS-109: Rename type argument plumbing to generic argument plumbing

## Description
Lane now treats explicit generic arguments as kind-aware arguments: Type-kind parameters receive type terms and Effect-kind parameters receive effect terms. Much of the compiler path is still named `type_arguments`, even when it carries `GenericArgument` values and validates Effect-kind arguments. The naming obscures the model and will get worse if more kinds are introduced.

## Design
Rename internal functions, fields, diagnostics labels, and helper modules where appropriate from type-argument terminology to generic-argument terminology. Keep user-facing source syntax and existing diagnostic wording stable unless a wording change makes the concept clearer. Avoid broad behavioral changes in this issue.

## Acceptance Criteria
- [ ] Kind-aware paths that carry `GenericArgument` are named generic arguments internally.
- [ ] Pure type-expression application code remains clearly separated from generic value/effect/owner application code.
- [ ] Public `.mbti` changes are intentional and reviewed.
- [ ] Tests demonstrate that Type-kind and Effect-kind explicit arguments still work after the rename.
