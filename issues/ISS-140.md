## Description
Lane should not treat `inf` and `nan` as language literals, but Basic should provide explicit `Double` constants that users can import when they need IEEE infinity or NaN values.

## Design
Add a `Basic.Double` module exporting ordinary values over `%f64_inf`, `%f64_ninf`, and `%f64_nan`:

```lane
module Basic.Double

pub let infinity : Double = builtin("%f64_inf")
pub let neg_infinity : Double = builtin("%f64_ninf")
pub let nan : Double = builtin("%f64_nan")
```

Extend the Lane runtime/interpreter builtin implementation for these constant intrinsic names. Examples should demonstrate the constants through ordinary Basic imports and existing comparison operations. `inf` and `nan` remain invalid as source literals.

Non-goals: `Double` printing/conversion helpers, total floating-point ordering, and implicit or explicit `Int`/`Double` conversions.

## Acceptance Criteria
- [ ] `../basic` contains `Basic.Double` with `infinity`, `neg_infinity`, and `nan`.
- [ ] Runtime/interpreter handles `%f64_inf`, `%f64_ninf`, and `%f64_nan`.
- [ ] Valid run examples cover infinity and NaN behavior using fixed success output.
- [ ] Invalid examples lock down that `inf` and `nan` are not source literals.
- [ ] Lane and `../basic` validation pass, including `tools/check-lane-run-examples.sh`.
