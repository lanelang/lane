## Description
Each handler `with` block must correspond to exactly one concrete effect, and
each handled operation must have useful and exhaustive payload-pattern arms.
The type checker must reject mixed-effect blocks, duplicate handled effects,
unknown operations, missing operation coverage, and non-useful payload arms.

## Design
Add checked handler operation-table construction. Group arms by concrete effect
owner, validate that each block handles one effect, reject duplicate handled
effects across blocks, and reuse the existing pattern matrix machinery to check
payload patterns per operation.

## Acceptance Criteria
- [ ] Checked handler data groups operation arms by concrete handled effect.
- [ ] A `with` block containing operations from multiple effects is rejected.
- [ ] A handler containing two `with` blocks for the same concrete effect is
  rejected.
- [ ] Every operation declared by a handled effect is covered by exhaustive
  payload patterns.
- [ ] Redundant or non-useful handler operation arms are rejected or diagnosed
  consistently with existing pattern usefulness diagnostics.
- [ ] Tests cover direct arms, payload-pattern arms, missing operation
  coverage, duplicate handled effects, and mixed-effect blocks.
- [ ] `moon check modules/lanec --warn-list +73` passes.
- [ ] `moon info && moon fmt` has been run after the change.
