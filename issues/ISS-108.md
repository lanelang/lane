# ISS-108: Parse member owners as nominal owner types

## Description
The parser currently parses nominal and effect owners through `postfix_expression`, then reconstructs a type/effect owner from expression-shaped syntax. This is visible in helper paths such as expression-to-type-name conversion for `Coroutine[Log]::yield!()` and `Box[Int]::{ ... }`. That design mixes value-expression grammar with type/effect owner grammar and relies on fallback strings for shapes that are not valid owners.

The correct language model is stricter: `T[X]` is already a complete type expression. In `T[X]::member`, the left side of `::` is not a value expression or a path with attached arguments; it is an owner type that must resolve to a nominal/effect owner accepted by the selected member form.

## Design
Parse the left side of `::` as a type-shaped owner, not as `Expr`. Reuse the existing type grammar where possible, but restrict the accepted forms to nominal/effect owner types either syntactically or during resolve/typecheck. The AST for struct literals, qualified variants, effect operation calls, and handler operation heads should carry an owner `Type` (or a small `NominalOwnerType` wrapper around `Type`) instead of splitting the owner into `type_name : String` plus `type_arguments`.

Preserve the source forms:

```lane
Box[Int]::{ value: 1 }
Option[Int]::some(1)
Coroutine[Log]::yield!()
Coroutine[Log]::spawn![A](body)
```

Reject or report non-owner type shapes such as function types or type lambdas before they can become empty owner names.

## Acceptance Criteria
- [x] Parser grammar has an explicit owner-type nonterminal for `Owner::member` forms and no longer reconstructs owners from arbitrary `Expr`.
- [x] Syntax AST represents member owners as complete type-shaped owner terms rather than `String` plus generic-argument arrays.
- [x] Existing struct literal, variant construction, effect operation call, and handler owner examples still parse and format unchanged.
- [x] Invalid owner type shapes produce targeted parser, resolver, or typechecker diagnostics instead of empty-name fallbacks.
- [x] Parser and formatter regression tests cover owner types with and without generic arguments.
