## Description
`CompileDiagnostic` currently carries only `message`, optional `span`, and `severity`. That shape forces every compiler phase to collapse rich context into a single string before CLI or LSP rendering. It is the main reason diagnostics look like internal enum names instead of user-facing explanations.

## Design
Add a structured report model owned by `lanec/compile` and usable by CLI/LSP:

- error code, e.g. `E0001`, with stable category allocation;
- severity;
- title/headline;
- primary label with span and text;
- zero or more secondary labels;
- notes for explanatory context;
- help entries for concrete fixes;
- optional machine-readable category/kind for tests and LSP.

Keep `span` for the primary source location, but avoid treating a diagnostic as a single string. Normal CLI rendering should consume the report model. LSP should expose the headline as `message` and place notes/help in fields supported by the protocol where practical.

## Acceptance Criteria
- [ ] `CompileDiagnostic` or its replacement can represent headline, labels, notes, help, and stable code.
- [ ] Existing parse/resolve/typecheck/module diagnostics can be converted without losing source spans.
- [ ] CLI rendering supports multi-line Rust-style output with source ranges.
- [ ] LSP diagnostics remain valid JSON and include at least code, severity, range, and a clear message.
- [ ] Existing callers are migrated directly; no permanent legacy string-only compatibility layer remains in user paths.
