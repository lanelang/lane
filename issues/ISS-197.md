## Problem Statement

Lane currently mixes several independent concepts in one runtime-I/O path. The Basic library declares console output as a source algebraic effect, command tooling recognizes the `Basic.Io.Write.println` shape, linking preserves runtime-effect-operation metadata, executable elaboration synthesizes root handlers, and the source `builtin` construct acts both as a compiler intrinsic and an open runtime plugin name. This makes effect semantics depend on module conventions, gives compiler-known operations and host-provided functions the same trust model, forces ordinary I/O through continuation machinery, and leaves duplicated runtime-effect metadata across artifacts and lowering stages.

The effect-erasure pipeline also needs a precise distinction between effects that change control flow and effects that only preserve observability. Treating every nonempty effect row as monadic would CPS-transform ordinary I/O, while erasing `Io` too early could make effect-sensitive optimization classify observable extern calls as pure.

## Solution

Lane will separate language effects, compiler intrinsics, extern linkage, runtime imports, and future host effect handling.

`Io` becomes an import-free built-in effect atom with intrinsic identity. It participates in effect rows, effect polymorphism, public interfaces, and purity analysis, but has no operations, cannot be handled, and does not occupy a nominal effect identity. It denotes observable interaction with the execution environment rather than the fact that an implementation happens to come from the runtime.

`builtin("...")` becomes a closed compiler-intrinsic construct. The compiler owns every accepted name, canonical type, effects, semantics, and lowering; unknown names and signature mismatches are compile errors. `extern("...")` becomes the open host-linkage construct. It requires a complete expected monomorphic function type, primitive host parameters and result, and a latent effect of `Empty` or exactly `Io`. The declaration is an unsafe programmer assertion and lowers to an ordinary LoisVM runtime import.

`Basic.Io` exports `println : (String) -> Unit ! Io = extern("%println")`. Neither the compiler nor command tooling recognizes `Basic.Io` or `%println` specially. Executable entries remain zero-argument functions returning `Unit` with an empty effect or exactly `Io`. The former runtime-effect-operation tables, Basic shape lookup, synthetic runtime root handlers, and runtime operation symbols are removed.

`mon-trans` uses the general monadic-effect predicate: an open effect row or any handled operation requires translation, and every handled operation is potentially multi-shot without resume-count analysis. Translation removes the monadic portion but preserves non-monadic residual effects such as `Io` on direct and CPS-transformed callable types. A final residual-effect-erasure pass runs only after effect-sensitive optimization and removes those type-level effects without deleting or reordering extern calls.

Extern calls remain synchronous primitive host calls without continuations. A future Host Effect Handler may receive algebraic-operation payloads and a first-class multi-shot continuation through a separate interface, but that capability is not part of this epic.

## User Stories

2. As a Lane programmer, I want `Io` to participate in ordinary effect rows, so that functions can expose and compose observable host interaction precisely.
3. As a Lane programmer, I want `Io` to participate in effect polymorphism, so that generic higher-order functions can be instantiated with I/O-producing callables.
4. As a Lane programmer, I want attempts to handle `Io` to fail statically, so that a non-handleable environmental effect is not confused with an algebraic operation set.
5. As a Lane programmer, I want a function carrying only `Io` to remain direct style, so that ordinary I/O does not pay continuation and dictionary overhead.
6. As a Lane programmer, I want a function carrying both `Io` and an algebraic effect to preserve `Io` after CPS conversion, so that optimization still observes its environmental interaction.
7. As a Lane programmer, I want `Basic.Io.println` to have the ordinary type `(String) -> Unit ! Io`, so that console output composes like any other typed function call.
8. As a Basic library author, I want to define `println` with `extern("%println")`, so that the Basic module contains no source effect declaration or operation wrapper for console output.
9. As a Basic library author, I want no compiler-recognized module path or exported shape, so that `Basic.Io` remains an ordinary replaceable library module.
10. As a Lane programmer, I want `builtin("...")` to mean a compiler-known intrinsic, so that it cannot silently resolve to arbitrary host behavior.
11. As a Lane programmer, I want an unknown builtin name to be a compile error, so that spelling mistakes cannot become unresolved runtime dependencies.
12. As a Lane programmer, I want builtin signatures checked against compiler-owned contracts, so that intrinsic types and effects cannot be forged with contextual annotations.
13. As a Lane programmer, I want `extern("...")` to mean an open host symbol, so that applications and embedders can add host functions without modifying the compiler.
14. As a Lane programmer, I want every extern to require a complete expected function type, so that its parameters, result, and effects are never inferred from incomplete call context.
15. As a Lane programmer, I want a missing extern effect annotation never to default to pure, so that observable calls cannot accidentally be optimized as pure computations.
16. As a Lane programmer, I want externs restricted to function types, so that host interaction always has an explicit invocation point and latent effect.
17. As a Lane programmer, I want an external value represented as a zero-argument function, so that reads from clocks, randomness, or environment state remain explicit and effectful.
18. As a Lane programmer, I want pure extern functions to be expressible, so that deterministic host-provided mathematics can remain pure under an unsafe declaration contract.
19. As a Lane programmer, I want observable extern functions to carry exactly `Io`, so that terminal, file, clock, randomness, environment, and network access share one environmental effect.
20. As a compiler maintainer, I want algebraic-effect, open-row, and effect-polymorphic extern signatures rejected, so that ordinary runtime imports never require handler contexts or continuations.
21. As a compiler maintainer, I want scalar, generic, and aggregate-ABI extern declarations rejected, so that the first extern ABI remains small and deterministic.
22. As a compiler maintainer, I want compiler intrinsics and extern bindings to have distinct IR origins, so that lowering never guesses from symbol strings.
23. As a compiler maintainer, I want extern bindings to lower directly to runtime imports, so that no synthetic effect operation, root handler, or adapter closure is required.
24. As a compiler maintainer, I want all handled operations treated as potentially multi-shot, so that correctness does not depend on incomplete resume-count analysis.
25. As a compiler maintainer, I want open effect rows to remain conservatively monadic, so that separately compiled effect-polymorphic code has a stable lowering contract.
26. As a compiler maintainer, I want non-monadic residual effects preserved through effect-sensitive optimization, so that CPS conversion cannot erase observability accidentally.
27. As a compiler maintainer, I want one explicit residual-effect-erasure pass, so that the transition from typed effects to effect-free ANF is reviewable and testable.
28. As a command user, I want `main` accepted when it is `() -> Unit` or `() -> Unit ! Io`, so that pure and I/O programs have one clear executable contract.
29. As a command user, I want residual algebraic effects rejected at entry selection, so that execution never depends on an implicit host handler.
30. As a command user, I want missing `%println` and other externs reported as load errors, so that host configuration failures occur before program execution.
31. As a command user, I want host invocation failures reported as fatal execution errors rather than Lane effects, so that the source effect system is not extended by backend failures.
32. As a LoisVM embedder, I want externs represented by the existing runtime-import function-table entries, so that custom hosts use one stable symbol registry and call model.
33. As a LoisVM embedder, I want runtime imports to remain synchronous, primitive-only, non-reentrant, and unable to retain VM values, so that ownership and execution-instance invariants stay tractable.
34. As a LoisVM embedder, I want pure and `Io` externs to share the same runtime ABI, so that source effects remain a compiler concern rather than a second host dispatch protocol.
35. As a backend maintainer, I want runtime imports treated as opaque observable calls after effect erasure, so that later lowering cannot delete or reorder host interaction.
36. As a backend maintainer, I want interpreter and Wasmoon execution to resolve the same symbols and produce the same output and failures, so that backend choice does not change Lane semantics.
37. As an artifact maintainer, I want runtime-effect-operation metadata removed from module, linked, and executable artifacts, so that artifacts describe actual executable dependencies only once.
38. As an artifact maintainer, I want affected binary schemas versioned rather than dual-decoded indefinitely, so that obsolete runtime-effect layouts do not remain permanent compatibility baggage.
39. As a project maintainer, I want examples migrated from `Write::println!` to `Basic.Io.println`, so that the complete corpus exercises the new public model.
40. As a project maintainer, I want canonical glossaries and ADRs to use only the new vocabulary, so that future work does not reintroduce runtime conventions or unsafe builtin plugins.

## Implementation Decisions

- Add `Io` as a dedicated built-in effect atom in source types and Buslane effects rather than reserving or synthesizing an effect identity.
- Make `Io` import-free, non-handleable, non-nominal, canonicalizable in unions, and visible in interfaces and artifacts wherever typed effects are preserved.
- Split external origins into compiler intrinsic, extern binding, and imported reference; do not classify by symbol spelling during late lowering.
- Keep `builtin("...")` as the source form for a closed compiler-owned intrinsic table with exact signature validation and no runtime fallback.
- Add `extern("...")` as the source form for open host symbols. It is checked only from a complete expected type and is represented explicitly through checked source, Buslane external metadata, linking, and executable lowering.
- Restrict externs to monomorphic function types over the primitive host ABI with latent effect `Empty` or `Io`. Reject scalar/global values, aggregate and closure values, generic parameters, effect parameters, open rows, and algebraic effects.
- Treat extern type and effect declarations as unsafe programmer assertions. The compiler uses them for optimization and lowering but does not infer host semantics from the symbol.
- Define `Basic.Io.println` as an ordinary exported extern-bound function. Remove the `Write` effect declaration, `Console`/`Io` aliases built from it, operation-call syntax, and compiler or command shape matching.
- Keep the command execution contract at zero parameters, `Unit` result, and residual effect `Empty` or `Io`.
- Remove runtime-effect-operation signatures and linked operation tables from compilation, fingerprinting, artifact codecs, linking, executable program models, command execution, and LoisVM lowering.
- Remove synthetic root runtime handlers and operation-derived runtime symbols. Runtime imports are created only from extern bindings.
- Define `is_mon_effect` independently of purity: open rows and rows containing any handled operation are monadic; `Io` alone is not.
- Perform no resume multiplicity analysis and no one-shot/multi-shot ABI specialization.
- Preserve the non-monadic residual effect on CPS-transformed callable and continuation types. For residual monadic contexts `M`, non-monadic effect `R`, and result `H`, use the conceptual residual computation `[Answer](context_arguments(M, Answer)..., continuation : (H) -> Answer ! R) -> Answer ! R`.
- Add residual effect erasure after `monadic-lift` and after every optimization that relies on effect rows. The pass changes type/effect metadata only and preserves calls, evaluation order, and control flow.
- Continue lowering externs to the unified LoisVM function table and Runtime Import ABI v1. Do not add a runtime-call opcode, continuation parameter, or second callable namespace.
- Keep runtime imports opaque after effect erasure. Pure extern optimization must happen while the source extern type is still available.
- Bump every persisted artifact schema whose runtime-effect table or effect encoding changes. Reject obsolete versions with the existing unsupported-version diagnostics instead of maintaining dual old/new semantics.
- Keep Host Effect Handler as a distinct future interface. It may eventually receive payloads and a multi-shot continuation, but it is not represented as an extern and is not implemented by this epic.

## Testing Decisions

- Use the production source-to-execution path as the primary seam: compile and link real Lane source, elaborate the selected executable, lower to LoisVM bytecode, execute through both the interpreter and Wasmoon tiers, and compare complete stdout and failure outcomes.
- Extend the existing effect-conformance style corpus rather than building a test-only lowering path. Cover direct I/O, I/O nested around algebraic handlers, mixed `Io` plus algebraic effects, higher-order and effect-polymorphic calls instantiated with `Io`, and multi-shot resume followed by I/O.
- Add compiler black-box tests for behavior that must fail before execution: unknown or mismatched builtins; externs without complete expected types; scalar, generic, aggregate, open-row, and algebraic-effect externs; attempts to handle `Io`; and invalid executable entry types.
- Add artifact roundtrip tests that inspect the complete public artifact value and prove that `Io` survives typed artifacts while runtime-effect-operation tables are absent. Cover explicit rejection of old schema versions.
- Add LoisVM public-interface tests that load an image containing `%println`, resolve it once, execute it, and report missing symbol, ABI mismatch, and host failure through existing load and execution errors.
- Run equivalent interpreter and Wasmoon tests for primitive arguments and results, String ownership, `Unit` results, runtime context sharing, cleanup after failure, and output ordering.
- Prefer `inspect` or `debug_inspect` for complete small structured values and diagnostics. Use direct assertions for stable scalar outcomes and avoid white-box tests when public interfaces are sufficient.
- Migrate the full example corpus and require every valid example to compile and run through the production command path with expected output.
- Finish with generated-interface, formatting, full test, warning, example, and diff checks required by the repository guidelines.

## Out of Scope

- A Host Effect Handler registry or ABI that exposes continuations to host code.
- VM re-entry, asynchronous externs, retained VM values, host callbacks, cancellation, or opaque host handles.
- Wasm GC, the Component Model, WASI Preview 2, or a second portable host ABI.
- Aggregate, closure, nominal-data, generic, variadic, multi-result, or effect-polymorphic extern signatures.
- Inferring extern signatures or effects from host registries, symbol names, call arguments, or return context.
- Resume-count analysis, one-shot continuation types, or handler-specific ABI cloning.
- Preserving compatibility with obsolete runtime-effect-operation artifact payloads beyond clear version rejection.
- Implementing additional Basic I/O functions beyond the migration needed to establish and test the new extern model.

## Further Notes

- Workspace ADR-0001 is the canonical architectural decision for this epic.
- The runtime import ABI remains an erased execution contract; `Io` is a source and typed-IR semantic effect, not a runtime capability object or dispatch tag.
- Historical runtime-effect and unsafe-builtin ADRs remain only as compact superseded pointers.
- The preferred implementation order is a vertical migration that keeps the production source-to-execution path buildable: establish type/IR vocabulary, add extern lowering, migrate Basic I/O and command execution, remove old metadata, then add residual effect erasure and complete artifact/example migration.
