## Description
Completion candidates now come from the unified semantic index, but cursor context is still classified by a collection of backward token scans and source-string offset helpers. The classifier manually recognizes imports, owner separators, type aliases, type/effect positions, and identifier prefixes. Every new syntax form can require another branch in completion independently of parser semantics.

## Design
Create one internal completion-syntax context module that accepts lexer tokens plus a cursor and returns a structured context such as import path, owner-qualified member, receiver selector, type position, effect position, or value position. Reuse parser-owned token and source-position utilities where possible. The module must remain tolerant of incomplete source; do not require a successfully parsed AST and do not move semantic candidate production back out of `SemanticIndex`.

## Acceptance Criteria
- [ ] Completion context classification has one interface and no scattered backward grammar scans in candidate production.
- [ ] Import, owner, receiver, type, effect, and value contexts are represented structurally.
- [ ] Identifier prefix and replacement spans use shared source-position utilities with UTF-8 behavior documented by tests.
- [ ] Incomplete and syntactically invalid source continues to receive useful completions.
- [ ] Adding a new declaration form does not require parallel completion-space heuristics outside the classifier.
- [ ] `moon info`, `moon fmt`, completion/LSP tests, full tests, and `git diff --check` pass.
