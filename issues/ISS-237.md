## Description
Unary and binary operators are mapped to ordinary Lane values such as `op_neg` and `op_add`. Resolver and desugar currently contain separate exhaustive mappings from the same operator enums to those names. A new operator can therefore resolve one target and desugar another if only one table is updated.

## Design
Resolve each operator target once and carry that semantic target into the resolved expression consumed by desugar. Keep operator sugar defined in terms of ordinary value resolution, including import provenance and unresolved-name diagnostics. Do not add a third standalone package or another shared string table merely to hide the duplication.

## Acceptance Criteria
- [ ] Each unary and binary operator target name is defined in exactly one production mapping.
- [ ] Resolved operator expressions carry enough information for desugar without recomputing the target name.
- [ ] Operator import-use tracking and unresolved-value diagnostics retain their current locations and behavior.
- [ ] Tests cover every operator and fail if resolver and desugar targets can diverge.
- [ ] `moon info`, `moon fmt`, focused tests, full tests, and `git diff --check` pass.
