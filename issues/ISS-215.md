# ISS-215: Emit the Reference layout for opaque imports

## Metadata
- Type: bug
- Status: closed
- Priority: 1
- Labels: area/lanec, area/loisvm, area/wasm, external-types, integration, tests
- Assignee: Codex

- Updated: 2026-07-16
- External ref: none

## Description
LoisVM lowering can emit `Opaque` Runtime Import signatures without emitting the canonical `Reference` layout required by their Lane ARC wrappers. The interpreter can execute such an image, but the Wasm compiler correctly rejects it with `Opaque runtime imports require a Reference layout`.

## Design
Treat every reachable `Opaque` Runtime Import parameter or result as a use of the canonical Reference layout during LoisVM layout planning. Do not weaken Wasm validation or synthesize backend-only layout metadata.

## Acceptance Criteria
- [x] Lowering an External Type extern guarantees one canonical Reference layout.
- [x] Programs without External Types retain their existing layout set.
- [x] The production External Type interpreter/Wasmoon conformance test passes.
- [x] Focused lowering and Lane tests pass.

- 2026-07-16: Discovered by compiling a source program whose only reference-shaped value was an External Type returned by a runtime import.

## Close Notes
Runtime-import planning now interns the canonical Reference layout whenever a reachable parameter or result kind is `Opaque`. The focused lowering regression asserts `[Reference]` for External Type imports and no layouts for primitive-only imports; the production source-to-interpreter/Wasmoon conformance test also passes.
