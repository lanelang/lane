# ISS-039: Add Lane effect syntax and symbol namespaces

## Metadata

## Description
Lane source needs first-class effect declarations and effect operation
identities before Buslane lowering can consume them. Today the syntax,
resolver, symbol registry, and module interface layers only model ordinary
types, values, fields, and variants.

## Design
Add source AST forms for `effect` declarations and operation declarations,
extend kind syntax with `Effect`, and introduce distinct effect constructor and
operation symbol namespaces. Resolution should keep operations out of the
ordinary value namespace and preserve source spans/presentation in `lanec`
structures only.

## Acceptance Criteria
- [ ] Syntax AST and parser represent effect declarations and operation
  signatures from the spec.
- [ ] Symbol registry has distinct effect constructor and operation identities.
- [ ] Resolver resolves effect constructors and operation names without using
  ordinary value lookup.
- [ ] Module interfaces can export/import effect declarations and operation
  signatures.
- [ ] Source spans and display names remain in `lanec`, not Buslane.
- [ ] Parser/resolver/module-interface tests cover effect declarations.
- [ ] `moon test modules/lanec` passes.
- [ ] `moon info && moon fmt` has been run after the change.
