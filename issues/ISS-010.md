## Description
Lane currently exposes many compiler, artifact, CLI, and LSP errors as terse internal strings such as `unresolved_value x`, `not_a_struct type#16`, `missing import Lib`, or `unexpected artifact token`. These messages are useful to compiler implementers but poor for users because they do not explain what failed, what source range matters, why the compiler expected something else, or what the user can do next.

Convert user-facing diagnostics to a Rust compiler inspired style: clear headline, error code, primary source label, secondary labels where useful, notes for cause/context, and actionable help. The goal is not to copy Rust formatting exactly, but to match its level of explanation and guidance.

Current message inventory:

- Syntax/format: `lexical error: illegal character ...`, `unterminated block comment`, `unterminated string`, `invalid string escape ...`, `invalid hex escape ...`, `non-ascii string character`, `parse error: unexpected ...`, `unexpected end of input`, `parser produced no source file`.
- Name/module resolution: `unresolved_type ...`, `unresolved_value ...`, `unresolved_field T.x`, `unresolved_variant T::v`, `payloadless_variant_call ...`, `conflicting_top_value ...`, `ambiguous_module_value ...`, `ambiguous_variant ...`.
- Type/kind/inference: `missing_top_level_let_annotation`, `type_arity_mismatch ...`, `type_alias_cycle ...`, `kind_mismatch ...`, `type_mismatch ...`, `not_callable ...`, `not_a_struct ...`, `invalid_int_literal ...`, `cannot_infer ...`.
- Struct/type-member construction: `unknown_field ...`, `missing_field ...`, `duplicate_field ...`, `unknown_type_member ...`, `missing_type_member ...`, `duplicate_type_member ...`, `duplicate_type_parameter ...`.
- Pattern/match: `non_exhaustive_match ...`, `unreachable_match_arm`, `duplicate_pattern_binder ...`, `pattern_arity_mismatch ...`, variant arity/owner failures currently hidden behind `cannot_infer ...`.
- Visibility/interface: `hidden_type_escape ...`, `private type ... in public interface`, stale or invalid interface fingerprint diagnostics, missing exported values/types/constructors/field selectors.
- Compile/link/artifact: `no root source module`, `duplicate module input ...`, `missing import ...`, `import cycle ...`, `unsupported module object artifact schema ...`, `module artifact fingerprint mismatch ...`, `invalid linked Buslane program`, `linked entry has unknown value ...`, `duplicate linked entry ...`.
- Artifact parser/text: `unexpected artifact token`, `unsupported artifact text version ...`, `expected tag ...`, `invalid Buslane block: ...`, `unknown Lane type tag ...`, `invalid optional value id ...`.
- CLI/LSP/runtime: `run target must be FILE:ENTRY`, `compile requires --emit-interface`, `entry not found: ...`, `ambiguous entry: ...`, `invalid hover params`, `invalid definition params`, interpreter runtime strings such as `callee is not callable`.

## Design
Use this epic as the umbrella. Implement the report model first, then migrate each diagnostic family without leaving compatibility-format rendering in normal user paths. Preserve structured data for tests and LSP. Do not add speculative recovery or suggestion engines beyond the concrete guidance required by each existing diagnostic.

## Acceptance Criteria
- [x] All child issues are closed.
- [x] `lane check`, `lane fmt`, `lane compile`, `lane link`, `lane run`, `lane runlinked`, and LSP diagnostics render user-facing diagnostics in the new style.
- [x] Internal enum/debug names no longer appear in normal user-facing errors unless explicitly marked as internal compiler bug details.
- [x] Golden tests cover representative diagnostics from syntax, resolve, typecheck, module/link/artifact, CLI, and LSP paths.
