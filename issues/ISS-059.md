## Description
`modules/lanec/compile/module_compile.mbt` still owns too many compiler workflow
concerns in one shallow module: module graph compilation, interface artifact
validation and ordering, single-module compilation, Buslane linking, imported
reference validation, lowered-object construction, and interface fingerprinting.

ISS-053 deepened Buslane identity remapping, but the surrounding compile
orchestration remains hard to reason about. The risk is that changes to
interface artifacts, import resolution, or linking can accidentally depend on
nearby workflow details instead of a phase-specific interface.

## Design
Split module compilation orchestration by phase. Candidate boundaries are:
module graph ordering, interface artifact validation/loading, single-module
front-end compilation, Buslane object linking, lowered object assembly, and
fingerprint calculation.

Each phase should expose the smallest result type needed by the next phase. The
split should preserve the existing public compiler API unless a separately
reviewed interface change is necessary.

Non-goals:
- Do not change artifact schema versions as part of this refactor.
- Do not change linker identity semantics already covered by ISS-053.
- Do not keep compatibility wrappers solely to preserve old internal call
  structure after migration.

## Acceptance Criteria
- [ ] Module graph compilation, interface artifact handling, linking, lowered-object assembly, and fingerprinting are separated behind phase-local interfaces.
- [ ] `module_compile.mbt` no longer acts as the primary implementation home for all compile workflow phases.
- [ ] Existing compile/link behavior is covered by tests before and after the split.
- [ ] `tools/check-lane-run-examples.sh` passes.
- [ ] `moon test modules/lanec/compile --warn-list +73` passes.
- [ ] `moon info && moon fmt` has been run after the change.
