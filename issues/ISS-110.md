# ISS-110: Make expression type and effect synthesis side-effect disciplined

## Description
The typechecker has paths that synthesize expression types, synthesize expression effects, build checked expressions, and emit diagnostics as interleaved operations. During explicit generic application work, effect synthesis needed to probe a callee type and then roll diagnostics back to avoid duplicate reports. That indicates the query-style paths are not side-effect disciplined.

## Design
Define a clearer contract for semantic queries: either build a checked expression once and reuse its type/effect, or provide explicit non-reporting probe APIs. Avoid using diagnostic rollback as normal control flow for effect computation. Preserve current type/effect behavior while reducing duplicate traversal and duplicate diagnostics.

## Acceptance Criteria
- [x] Effect synthesis no longer relies on diagnostic checkpoint/rollback to avoid duplicate reports.
- [x] Call checking computes callee type, argument effects, and latent effect through a shared checked or non-reporting path.
- [x] Regression tests cover explicit generic application error paths without duplicate diagnostics.
- [x] Existing effectful call and handler tests still pass.
