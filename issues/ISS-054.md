# ISS-054: Encode Buslane lowering registration invariants

## Description
`modules/lanec/elaborate/buslane_lower.mbt` currently relies on a large
implicit ordering invariant: source symbols must have been registered in the
Buslane lowerer before any lowering code asks for their Buslane ids. When the
invariant fails, helper methods such as `type_id`, `value_id`, `effect_id`, and
`operation_id` abort with messages like `unknown Buslane value id`.

This makes lowering fragile as new source features are added. A future
maintainer must know the exact predeclaration and alias registration order for
types, values, fields, variants, effects, and operations.

## Design
Move registration invariants behind a deeper lowering environment interface.
The environment should make each phase explicit: declaration prepass,
metadata/owner registration, expression lowering, and imported reference
binding. Missing ids should either be impossible by construction or reported as
structured compiler diagnostics at the boundary, not arbitrary aborts deep in
expression lowering.

Non-goals:
- Do not remove assertion-style aborts for truly impossible internal corruption
  until the registration flow has a stronger seam.
- Do not change Buslane's identity model.

## Acceptance Criteria
- [ ] Buslane lowering has an explicit registration phase interface for nominal types, values, effects, operations, and generated helper values.
- [ ] Most `unknown Buslane ... id` aborts in the lowerer are removed or moved behind one checked invariant boundary.
- [ ] Tests cover at least one effect declaration, one nominal declaration, and one imported reference through the new seam.
- [ ] Existing elaborate snapshots remain semantically equivalent or are updated with reviewed diffs.
- [ ] `moon test modules/lanec/elaborate --warn-list +73` passes.
- [ ] `moon info && moon fmt` has been run after the change.
