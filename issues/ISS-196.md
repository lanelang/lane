# ISS-196: Make abstract residual contexts safe across CPS answer changes

## Metadata

## Description
ADR-0067 models an abstract effect parameter as one ordinary companion type/value carrying an opaque context. That context is tied to the Answer type at which its concrete handler dictionary was created. A handler with abstract residual effect `E` changes the handled body's local answer to `M_E<H>` and currently reuses the outer opaque `E` context at that different answer. This is not type- or semantics-preserving.

Example 37 exposes the defect in `run_task[E]`: a handled Coroutine clause must return `M_E<Step>`, but execution returns the outer `Int -> RunResult` answer and later calls it through the residual callable ABI, producing an indirect-call type mismatch in both interpreter diagnostics and Wasm.

## Design
Reopen the ADR-0067 representation decision before implementation. The replacement must let an abstract residual context be adapted to a fresh CPS Answer without runtime effect tags or LoisVM effect instructions. Candidate directions include whole-program effect specialization before selective CPS, or an answer-polymorphic compiler-private context family carrying enough ordinary runtime evidence to materialize/relay the context at each Answer. Do not accept a callable adapter that merely hides the mismatch; the residual clause must actually produce `M_E<H>`.
