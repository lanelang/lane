# Position-specific semantic completion

Lane semantic completion is a position-specific compiler analysis query rather than a precomputed field of `AnalysisResult`. Ordinary analysis keeps diagnostics, navigation, hovers, semantic tokens, and inlay hints, while completion computes candidates for one cursor location from the same source inputs and completion trigger context. This avoids storing scope snapshots for every AST node, keeps normal analysis results compact, and preserves the boundary where LSP completion is only a protocol adapter over compiler-owned semantic completion.

Completion should be more tolerant than diagnostics without inventing a second language semantics. Resolve-level lexical and module candidates may still be returned when type checking fails, but receiver completion such as `expr.` requires a checked receiver type and returns no typed field/member candidates when that type is unavailable. Parser recovery for completion should stay narrow in v1 and cover only simple cursor contexts such as an identifier prefix or qualified access prefix.

Completion visibility follows Lane visibility rules. Current-module completion may expose locally visible private declarations, but imported module completion only exposes the module interface surface, owner-visible members, and visible offers. Completion must not use module objects or linked artifacts to reveal private lowered definitions. Ambiguous names remain resolver diagnostics rather than a separate completion error model in v1.

The compiler owns prefix filtering, replacement ranges, stable ordering, and semantic deduplication. V1 should prefer exact and prefix matches, then local lexical candidates before imported or module-surface candidates, while adapting kind priority to obvious value, type, or effect positions. Entries with the same semantic symbol are deduplicated, but different symbols with the same label remain separate candidates with distinguishing detail rather than being silently dropped.

Completion context classification should be explicit. V1 distinguishes ordinary identifier completion, module-qualified completion, nominal-qualified constructor or variant completion, receiver member completion, effect operation completion, and unsupported contexts. Because `.` can mean module-qualified access or receiver access, a visible module binding on the left side takes priority over treating the same text as an expression receiver.

The `::` completion context is limited to owner-defined named members. Nominal type owners complete variants or constructor-like members according to the source language, effect owners complete operations, and module-qualified owners such as `Lib.Option::` or `Lib.Console::` use only the imported module interface surface. `Lib::x` is not a module-member completion form because module access uses `.`.

Receiver completion for `expr.` is selector completion, not method completion. Lane has no method concept, so v1 completes only source-language selectors such as struct fields and any existing type-member access forms supported by the language. Ordinary functions are never suggested after `expr.` merely because their first parameter type matches the receiver.

Import declaration completion is part of v1 semantic completion, but only over module paths already present in the module input set or workspace analysis inputs. Completion must not introduce filesystem search by module name, must not suggest the root module as an import candidate, and must not auto-insert imports in v1.

The LSP adapter advertises semantic completion with trigger characters `.` and `:`, maps compiler `CompletionEntry` values into LSP completion items, and returns a `CompletionList` with `isIncomplete=false`. V1 completion does not use snippet insert text or resolve-provider follow-up requests. The compiler still interprets `::` contexts by inspecting the source around a `:` trigger because LSP trigger characters are single characters.

Completion tests are split by ownership. Compiler tests cover semantic candidate selection, scope and shadowing, type/effect/import contexts, module-qualified and `::` owner contexts, checked selector completion, and partial-analysis behavior. LSP tests cover capability advertisement, request parameter validation, completion list transport, trigger-character routing, and text-edit range mapping without duplicating every compiler semantic case.
