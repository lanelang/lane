## Description
Lane source needs to parse and print handler expressions before semantic
checking can attach operation identities or effects. The syntax must represent
the handled expression, one or more `with` blocks, operation arms, payload
patterns, resume binders, and the required `final` branch.

## Design
Add source AST nodes for handler expressions and update the parser, lexer
keywords if needed, source pretty printer, and parser snapshots. Keep this
change source-only: do not invent resolved, desugared, checked, or Buslane
handler data in this issue.

## Acceptance Criteria
- [x] Source AST represents handler body, `with` blocks, operation arms,
  payload patterns, resume binders, and final branch.
- [x] Parser accepts the spec grammar for handler expressions, including
  zero-payload arms whose only parameter is the resume binder.
- [x] Parser rejects malformed handler forms through existing parser error
  paths.
- [x] Source pretty printing renders handler expressions deterministically.
- [x] Parser or syntax tests cover direct arms, payload-pattern arms, and final
  branches.
- [x] `moon check modules/lanec --warn-list +73` passes.
- [x] `moon info && moon fmt` has been run after the change.
