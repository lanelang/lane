## Description
Compile diagnostic rendering still contains string-content heuristics for
choosing inference help text. `cannot_infer_help(label)` inspects whether a
rendered label contains phrases such as `contextual offer`, `function literal
parameter`, `arity`, `type arguments`, or `builtin`.

This makes structured diagnostics depend on presentation text. Changing a
typechecker label can silently change or degrade user-facing help, and tests may
only catch the final text after the structured cause has already been lost.

## Design
Carry an explicit inference diagnostic reason from type checking into compile
diagnostics. Rendering should switch on the structured reason, not on the
display label. Keep fallback text for generic inference failures, but avoid
semantic decisions based on `String::contains`.

Non-goals:
- Do not redesign all type diagnostics in this issue.
- Do not remove user-facing help text; preserve or improve the current messages.

## Acceptance Criteria
- [ ] `cannot_infer_help` no longer uses substring matching to select semantic help.
- [ ] Typecheck/compile diagnostics carry a structured reason for the known inference cases currently detected by label text.
- [ ] Golden diagnostics cover contextual offers, function literal parameters, arity, type arguments, and builtin inference failures.
- [ ] `moon test modules/lanec/compile/diagnostic_golden_wbtest.mbt --warn-list +73` passes.
- [ ] `moon info && moon fmt` has been run after the change.
