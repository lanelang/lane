## Description
Support transparent aliases for effect sets so Lane programs can name effect combinations with top-level `type` aliases, such as `type UI = { Console, Log }`, and use them in function effect annotations such as `fn f() -> Unit ! UI`.

## Design
Use the term Effect Set Alias. Reuse top-level type aliases instead of adding a new effect-alias declaration form. Alias bodies may have effect kind, support effect set literals and Effect-kind parameters, expand transparently during typechecking, normalize to ordinary effect sets, and do not survive into Buslane. Public effect set aliases remain transparent `type_aliases` in module interfaces.

## Acceptance Criteria
- [x] Lane accepts simple and generic effect set aliases in effect annotations.
- [x] Effect set aliases are transparent, normalized, and alias-free after checking/lowering.
- [x] Public effect set aliases roundtrip through module interfaces and artifact inspection.
- [x] Invalid cases cover private effect leaks, kind mismatch, cycles, and duplicate effect/type names.
