# ISS-240: Separate coverage witnesses from checked source patterns

## Description
Pattern exhaustiveness witness generation constructs synthetic `CheckedPattern` trees and fills every source location with `SourceSpan::dummy()`. This reuses a source-bearing semantic AST for generated coverage terms, allowing synthetic locations to leak into later logic and obscuring which pattern operations require real source provenance. Missing-witness rendering also embeds an unexplained recursion depth of six.

## Design
Give the coverage algorithm a private, span-free pattern or constructor representation designed for matrix usefulness and witness rendering. Convert source `CheckedPattern` values into that representation at the coverage seam instead of manufacturing checked source nodes. Name and document the witness-expansion budget; reaching it may abbreviate presentation but must not change exhaustiveness correctness.

## Acceptance Criteria
- [ ] Coverage candidate generation does not construct `CheckedPattern` values with dummy spans.
- [ ] Coverage usefulness and witness rendering use a private span-free representation.
- [ ] The witness-expansion budget is named, documented, and tested on deeply nested nominal types.
- [ ] Exhaustiveness and redundancy decisions are unchanged when witness rendering reaches its budget.
- [ ] User-facing missing-pattern diagnostics remain readable and source-located at the owning match or declaration.
- [ ] `moon info`, `moon fmt`, focused tests, full tests, and `git diff --check` pass.
