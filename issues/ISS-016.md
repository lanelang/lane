## Description
Pattern and match diagnostics are currently too terse for a language feature where users need guidance. Examples include `non_exhaustive_match Option[Int]`, `unreachable_match_arm`, `duplicate_pattern_binder x`, `pattern_arity_mismatch some expected=1 actual=2`, and variant-related `cannot_infer ...` messages.

Current message inventory:

- `non_exhaustive_match ...`
- `unreachable_match_arm`
- `duplicate_pattern_binder ...`
- `pattern_arity_mismatch ... expected=... actual=...`
- variant owner/arity/hidden type argument errors currently represented as `cannot_infer ...`

## Design
Use pattern matrix information to explain non-exhaustiveness with at least one missing shape when available. For unreachable arms, label the previous covering arm if the checker can identify it. For variant arity and owner failures, explain variant constructor syntax, payload count, and the enum owner if known.

## Acceptance Criteria
- [ ] Non-exhaustive match diagnostics include the scrutinee type and at least one missing pattern example where feasible.
- [ ] Unreachable arm diagnostics explain that an earlier pattern already covers this arm.
- [ ] Duplicate binder diagnostics explain that Lane patterns do not imply equality constraints.
- [ ] Variant arity diagnostics show expected versus actual payload count and suggest the correct constructor shape.
- [ ] Golden tests cover nested patterns, wildcard coverage, unreachable arms, and constructor arity.
