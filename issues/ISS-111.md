## Description
Adding a new expression node requires updating multiple independent traversal sites: syntax pretty printing, resolver, desugarer, checked construction, source analysis, annotation collection, completion, and related LSP analysis paths. Some missing traversal cases do not fail compilation immediately, but can silently degrade hover, go-to-definition, references, inlay hints, or completion.

## Design
Introduce traversal helpers or visitor-style APIs at the appropriate layers so new expression nodes have a smaller and more auditable update surface. Keep layer-specific logic local, but make child traversal systematic enough that LSP and analysis paths cannot silently miss new nodes.

## Acceptance Criteria
- [ ] Resolved and checked expression traversal has a shared, explicit child-walking utility or equivalent pattern.
- [ ] LSP analysis paths for hover, definition, references, inlay hints, and completion are covered by tests when new expression nodes are introduced.
- [ ] Existing `TypeApplication` traversal is represented in the shared path rather than copied in every analysis function.
- [ ] No behavior regressions in existing LSP tests.
