## Description
Several diagnostics are weak because parser and typechecker code reports the immediate failure but drops the reason that made a token, type, effect, or operation expected. Without expectation provenance, the final report cannot explain the real cause.

## Design
Thread expectation causes through parser recovery, type inference, effect checking, handler checking, and name resolution. The compiler should know not only what failed, but which syntax context, type annotation, call argument, handler arm, or declaration introduced the expectation.

## Acceptance Criteria
- [x] Parser diagnostics can identify the enclosing syntactic expectation when it matters.
- [x] Type mismatch diagnostics can point to the source of the expected type and the expression producing the actual type.
- [x] Effect and handler diagnostics can point to the operation/effect declaration or handler arm that introduced the constraint.
- [x] Regression tests demonstrate at least one improved parser diagnostic and one improved type/effect diagnostic.
