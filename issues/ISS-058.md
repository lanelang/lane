# ISS-058: Split Buslane lowering into local phases

## Description
`modules/lanec/elaborate/buslane_lower.mbt` is still a large locality problem.
Even after registration invariants were moved behind `BuslaneLoweringRegistry`,
the file mixes imported interface seeding, declaration collection, module export
construction, expression lowering, handler lowering, pattern-decision lowering,
metadata generation, and symbol definition helpers.

This makes the lowering implementation hard to modify safely. A maintainer
working on handlers or pattern lowering must keep unrelated import and metadata
ordering rules in their head because they live in the same implementation module.

## Design
Split Buslane lowering into phase-local modules with narrow interfaces. Keep the
existing Buslane output model, but separate at least declaration/import
registration, expression lowering, handler lowering, pattern-decision lowering,
and export/metadata construction.

The split should preserve the explicit registry seam from ISS-054. The goal is
to improve locality and make each phase testable without reintroducing hidden
ordering dependencies.

Non-goals:
- Do not change Buslane semantics or textual output as part of the split.
- Do not leave compatibility wrappers that preserve the old monolithic
  structure after callers are migrated.

## Acceptance Criteria
- [ ] Buslane lowering is split into phase-local implementation files or modules with explicit interfaces.
- [ ] Handler lowering and pattern-decision lowering no longer live in the same large file as import seeding and export construction.
- [ ] The `BuslaneLoweringRegistry` seam remains the single place for lowering identity lookup and registration.
- [ ] Existing elaborate and compile tests remain semantically equivalent or snapshots are reviewed and updated.
- [ ] `moon test modules/lanec/elaborate --warn-list +73` passes.
- [ ] `moon test modules/lanec/compile --warn-list +73` passes.
- [ ] `moon info && moon fmt` has been run after the change.
