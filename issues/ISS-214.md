# ISS-214: Accept external effects on executable entries

## Description
Executable-entry validation still accepts only `Empty` and built-in `Io`, even though user-declared External Effects use the same direct residual-effect model and the canonical design permits any closed row containing only External Effects.

## Design
Validate entry effects against Buslane effect metadata. Accept `Empty`, `Io`, and fully concrete singleton or union terms whose owning effect flavor is `External`; continue rejecting effect parameters, unknown effects, and Algebraic Effects. Apply the same rule in executable elaboration and Lane CLI preflight diagnostics.

## Acceptance Criteria
- [x] Whole-program executable elaboration accepts a closed user External Effect entry.
- [x] Lane CLI entry preflight accepts the same entry and still rejects Algebraic Effects.
- [x] Entry diagnostics describe the general External Effect contract rather than only `Io`.
- [x] Focused executable and Lane tests pass.

## Close Notes
Centralized executable-entry effect policy in `lanec/executable`, using Buslane effect flavor metadata to accept concrete External Effect rows and reject Algebraic or parameterized effects. Lane run/link preflight now reuses the same policy, and diagnostics describe the general External Effect contract. Verified with focused executable and Lane entry tests, package checks, `moon info`, formatting checks, and `git diff --check`.
