## Description
The `.lmi`, `.lmo`, and `.lbp` text artifact parser has many low-level failure strings. Users will see these when passing corrupted, hand-edited, or stale artifacts to `lane compile`, `lane link`, or `lane runlinked`.

Current message inventory:

- `unexpected end of artifact`
- `unexpected artifact token`
- `unsupported artifact text version ...`
- `unknown artifact kind ...`
- `expected interface/object/program artifact, got ...`
- `unexpected trailing artifact entry`
- `expected tag ...`
- `expected atom/string field ...`
- `invalid integer for ...`
- `invalid boolean for ...`
- `invalid Buslane block: ...`
- `expected buslane block, got ...`
- `unknown interface type tag ...`
- `expected tag-only entry, got ...`
- `expected value parameters, got ...`
- `unknown interface value kind ...`
- `unknown Lane kind ...`
- `unknown Lane type tag ...`
- `unknown Lane primitive type ...`
- `invalid Buslane type for ...`
- `invalid optional type/data constructor/value id for ...`
- `unknown export kind ...`

## Design
Artifact parser diagnostics should still be concise, but they need enough context to tell the user whether they passed the wrong file kind, edited an artifact incorrectly, or used an incompatible compiler version. Use artifact field names as labels and preserve parseable structured errors for tests.

## Acceptance Criteria
- [x] Wrong artifact kind/version errors explain the expected file kind and likely command that produced it.
- [x] Field/tag errors identify the artifact section and expected field.
- [x] Embedded Buslane parse errors include a nested note rather than a flat string.
- [x] CLI callers preserve the artifact path around parser errors.
- [x] Golden tests cover wrong kind, wrong version, missing field, invalid integer, and invalid embedded Buslane.
