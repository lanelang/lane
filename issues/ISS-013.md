## Description
Resolver diagnostics expose internal snake_case names and often omit the relevant import/module context. Examples include `unresolved_value Core.Int.id`, `ambiguous_module_value core.int.add`, and `payloadless_variant_call none`.

Current message inventory:

- `unresolved_type ...`
- `unresolved_value ...`
- `unresolved_field Type.field`
- `unresolved_variant Type::variant`
- `payloadless_variant_call ...`
- `conflicting_top_value ... existing=... incoming=...`
- `ambiguous_module_value ...`
- `ambiguous_variant ... [Owner::variant, ...]`

## Design
Convert resolver diagnostics to explain the failed lookup domain: type namespace, value namespace, field lookup, variant lookup, module-qualified value lookup, or import conflict. Diagnostics should say which names are in scope when useful, whether a module was imported, and whether the user probably meant a variant constructor, a module-qualified name, or a selective import.

## Acceptance Criteria
- [ ] Unresolved type/value diagnostics name the lookup namespace and show the exact unresolved spelling.
- [ ] Ambiguity diagnostics list candidate modules/types and tell the user to qualify the reference or adjust imports.
- [ ] Payloadless variant call diagnostics explain that payloadless variants are values and should be used without call syntax.
- [ ] Conflicting top-level value diagnostics point to both declarations when both spans are available.
- [ ] Golden tests cover unresolved names, ambiguous imports, payloadless variants, and conflicts.
