## Description
Type checker diagnostics currently expose messages such as `type_mismatch expected=Int actual=Bool`, `kind_mismatch expected=Type actual=[Type] -> Type`, `not_callable Int`, `not_a_struct type#16`, and broad `cannot_infer ...` labels. These should become explanatory diagnostics that state what expression was checked, what type/kind was expected, what was found, and how to make the program typecheck.

Current message inventory:

- `missing_top_level_let_annotation`
- `type_arity_mismatch ... expected=... actual=...`
- `type_alias_cycle ...`
- `kind_mismatch expected=... actual=...`
- `type_mismatch expected=... actual=...`
- `not_callable ...`
- `not_a_struct ...`
- `invalid_int_literal ...`
- `cannot_infer builtin`, `cannot_infer call arity`, `cannot_infer type arguments`, `cannot_infer function literal parameter`, `cannot_infer missing contextual offer`, `cannot_infer ambiguous contextual offer`, and other context labels.

## Design
Keep expected/actual data structured instead of embedding it in a string. Replace `CannotInferExpression(String)` catch-all cases with specific diagnostic kinds where the user action differs. For `not_a_struct type#...`, prefer showing the source-level type when available and add a note if only an internal type id is known.

## Acceptance Criteria
- [ ] Type mismatch diagnostics show expected and actual types in labelled notes or secondary spans.
- [ ] Kind mismatch diagnostics explain `Type` versus higher-kinded forms and suggest adding or correcting kind annotations where applicable.
- [ ] Arity diagnostics name the callee/type constructor and show expected versus provided counts.
- [ ] `CannotInferExpression` catch-all user paths are split or wrapped with actionable help.
- [ ] Golden tests cover type mismatch, kind mismatch, not callable, not struct, arity, contextual offer, invalid int, and type alias cycle diagnostics.
