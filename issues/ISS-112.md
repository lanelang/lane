## Description
Generic expression application diagnostics are currently adapted from type constructor and generic function inference diagnostics. For example, arity mismatches use a generic `generic expression` subject, and empty `[]` is reported as a generic parser unexpected-token error. These messages are correct enough, but they are not as precise as the language feature deserves.

## Design
Add diagnostic shapes and parser expectations that understand explicit generic expression application as a first-class construct. Error messages should identify the callee expression where possible, distinguish empty generic argument lists from unrelated parse failures, and explain Effect-kind vs Type-kind mismatches in terms of explicit generic arguments.

## Acceptance Criteria
- [ ] Arity mismatch for `expr[Args]` reports a precise generic application subject and expected/actual argument counts.
- [ ] Empty `expr[]` produces a targeted diagnostic explaining that empty generic argument lists are not allowed.
- [ ] Kind mismatch for Effect-kind generic arguments points at the offending argument and explains the expected kind.
- [ ] Non-generic callee diagnostics remain distinct from callability diagnostics.
- [ ] CLI fixture tests lock the improved diagnostic output.
