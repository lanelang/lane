## Description
Convert declaration heads and expression chains to use width-sensitive formatting after list-like syntax is covered.

## Design
Cover `let` and `offer` initializers, function and effect operation signatures, function literals, `if` heads, `match` heads, `handle` heads, binary operator chains, pipeline chains, and field-access chains. Broken assignments move the initializer to the next line with two-space indentation. Broken binary, pipeline, and field-access continuations put the operator, `|>`, or `.` at the start of the continuation line.

## Acceptance Criteria
- [ ] Long `let` and `offer` initializers format with the initializer on a continuation line.
- [ ] Function, function literal, and effect operation signatures prefer broken parameter lists before breaking return/effect tails.
- [ ] Long `if`, `match`, and `handle` heads break before their controlling expression without losing structural clarity.
- [ ] Binary, pipeline, and field-access chains use operator-leading continuations when broken.
- [ ] Formatter output remains parseable and idempotent for each covered shape.
- [ ] `moon test modules/lanec/syntax --warn-list +73` and `moon test modules/lanec/format --warn-list +73` pass.
