# ISS-108: Separate parser owner syntax from value expressions

## Description
The parser currently parses nominal and effect owners through `postfix_expression`, then reconstructs a type/effect owner from expression-shaped syntax. This is visible in helper paths such as expression-to-type-name conversion for `Coroutine[Log]::yield!()` and `Box[Int]::{ ... }`. That design mixes value-expression grammar with type/effect owner grammar and relies on fallback strings for shapes that are not valid owners.

## Design
Introduce an explicit owner syntax model for nominal/effect owners used before `::`, instead of recovering owners from arbitrary expressions. Keep value-level field access and type/effect-qualified owner paths distinct in the parser and syntax AST. The refactor should preserve existing accepted source forms while removing expression-to-type-name fallback behavior.

## Acceptance Criteria
- [ ] Parser grammar has an explicit owner nonterminal for `Owner::member` forms and no longer reconstructs owners from arbitrary `Expr`.
- [ ] Syntax AST represents owner paths and owner generic arguments without passing through value-expression nodes.
- [ ] Existing struct literal, variant construction, effect operation call, and handler owner examples still parse and format unchanged.
- [ ] Invalid owner shapes produce targeted parser or resolver diagnostics instead of empty-name fallbacks.
- [ ] Parser and formatter regression tests cover owner paths with and without generic arguments.
