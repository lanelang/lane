## Description
Many diagnostics are currently emitted as a headline plus string notes and helps. Rust-style diagnostics need structured facts: expected versus actual values, where an expectation came from, which rule produced a constraint, and which source locations are relevant.

## Design
Extend the diagnostic data model or diagnostic-builder layer with typed cause information that compiler stages can attach before rendering. Keep terminal rendering, LSP transport, and JSON output as consumers of this structure rather than owners of diagnostic meaning.

## Acceptance Criteria
- [x] There is a clear representation for expected/actual facts, causal notes, and source-backed related locations.
- [x] The representation can be rendered by CLI and transported through LSP/JSON without lossy string parsing.
- [x] At least one high-value diagnostic uses the structured representation end to end.
