# ISS-096: Separate CLI command reports from source diagnostics

## Description
Lane CLI errors currently render many command, artifact, runtime, and entry-protocol failures with a Rust-like `= label`, `= note`, and `= help` layout even when there is no source span. This makes command errors look like incomplete source diagnostics and creates ad-hoc hand-written report formatting in `CliError::render_human`.

## Design
Introduce a structured command-report model for CLI-only failures. Keep source diagnostics on the existing `Milky2018/diagnostic` path, where labels and snippets are meaningful. Convert `CliError` into command reports and render them with a CLI-oriented renderer that uses fields, paragraphs, and suggestions instead of fake source labels. Do not change compile diagnostic or LSP diagnostic semantics in this issue.

## Acceptance Criteria
- [x] CLI-only errors no longer render fake `= label`, `= note`, or `= help` lines.
- [x] `CliError::render_human` delegates to a structured command report renderer instead of assembling every report directly.
- [x] Existing compile diagnostic rendering continues to use source diagnostic rendering.
- [x] Tests cover representative command errors, including invalid run targets, missing entries, non-executable entries, duplicate objects, root library duplication, and internal errors.
- [x] Documentation records the boundary between source diagnostics and command reports.
