## Description
Diagnostic color should not be a hardcoded renderer detail. Lane needs a clear terminal capability policy covering when to emit ANSI color, how to disable it, how to force it, and how test snapshots stay stable.

## Design
Introduce a small output configuration layer for diagnostics. Use plain output by default in tests and non-terminal sinks. For interactive terminals, prefer auto color when stderr/stdout is a TTY and `NO_COLOR` is not set. Support explicit flags or environment overrides for `always`, `never`, and `auto`. Keep LSP and JSON outputs color-free.

## Acceptance Criteria
- [x] Diagnostic rendering has an explicit color mode: auto, always, or never.
- [x] Auto mode respects TTY detection and `NO_COLOR`.
- [x] CLI tests can force plain output deterministically.
- [x] Documentation states that LSP/JSON diagnostics never include ANSI escapes.

## Notes
- 2026-07-03: Candidate implementation can use `moonbitlang/x/sys` or another MoonBit system package if it exposes TTY/environment primitives; if not, add a minimal native-only terminal capability shim local to the CLI boundary.
- 2026-07-03: For native TTY FFI shape, reference `Milky2018/jarvis` `src/readline/tty.c`: https://github.com/Milky2018/jarvis/blob/main/src/readline/tty.c. It uses MoonBit C FFI exports, `ioctl(TIOCGWINSZ)` for terminal size, `termios` for raw mode, and errno-style integer error returns. Lane diagnostics likely need a smaller subset: TTY detection, terminal width, and environment/color policy, not raw input mode.
