# ISS-115: Warn on unused local value bindings

- Status: closed

- Assignee: codex
- Created: 2026-07-05

## Description
Lane should emit a warning when a local value binding is introduced but never used. This covers value-level binders inside executable bodies, including local lets, pattern binders, function and lambda parameters, local function declarations, and handler binders. The goal is to catch likely mistakes without turning warnings into compile errors.

## Design
Define this as an unused local value binding warning, not as dead-code elimination or API reachability. Add a checked-AST value-use analysis that records local value binders and resolved `ValueSymbolId` references after source elaboration. Each reference should carry its origin binder when the reference occurs inside a local binder's definition body, and no origin otherwise. This analysis is a symbol identity graph, not a lexical scope tree: resolved symbols already encode binding identity and shadowing. A local value binding counts as used when its resolved symbol is referenced from outside its own definition body. For example, in `let x = 1; let y = x; 0`, `x` is used because `y`'s initializer references it, while `y` is still unused. Function and lambda parameters do not have their own definition body; they count as used when referenced anywhere in their function body, including nested local functions. Pattern binders also do not have their own definition body; they count as used only when referenced in the region where the pattern binding is visible. A pattern-let initializer, a match scrutinee, or a handler operation payload expression does not count as a use of the binder introduced by the pattern. A local function that captures `x` also makes `x` used, even if the local function itself is unused; the function binding receives its own warning instead. Local recursive groups should be handled by strongly connected components over the binder dependency graph: a component is used only when it has an incoming reference from outside the component, including a reference from an ordinary expression with no origin binder. A local function referenced only by its own recursive body still warns, and mutually recursive local functions with no reference from outside the recursive group also warn. Do not try to determine whether a reference contributes to the final result; that belongs to future reachability/DCE work. Local function declarations are ordinary local value binders for this rule. The reusable analysis should stay policy-free enough for future DCE to consume its symbol-use facts; warning policy such as `_x` suppression belongs in the unused-warning consumer. A named local binding whose source name starts with `_` is intentionally ignored and does not produce this warning. Run the analysis only after source elaboration/typecheck succeeds and a complete checked AST is available; do not emit these warnings on failed typecheck results in v1. Warnings are diagnostic output only: they must not make `check`, `compile`, `run`, or `runobj` fail when there are no errors, and they must not block artifact generation or execution. Public API declarations and private top-level declarations are out of scope for this issue.

Use diagnostic code `W2001`. The diagnostic message should be `unused local value binding`, the primary label should say `` `name` is never used ``, and the help should say `prefix the name with `_` if this is intentional`.
Emit unused local warnings in binder source-location order so CLI, LSP, and fixture output remains stable.

## Acceptance Criteria
- [x] Unused local `let` bindings produce warning diagnostics at the binder location.
- [x] A local binding read by a later local initializer counts as used even when the later binding is unused.
- [x] Unused local function declarations produce warning diagnostics at the function binder location.
- [x] A local function referenced only by its own recursive body still produces this warning.
- [x] Mutually recursive local functions with no reference from outside the recursive group still produce warnings.
- [x] Recursive local binder groups are classified using strongly connected components over binder dependency edges.
- [x] Unused pattern binders and function/lambda parameters produce warning diagnostics at the binder location.
- [x] Function and lambda parameters count as used when referenced anywhere in their function body, including nested local functions.
- [x] Pattern binders count as used only when referenced in the region where the pattern binding is visible.
- [x] Unused handler final/resume/payload binders produce warning diagnostics at the binder location.
- [x] Handler resume binders count as used only when referenced in the corresponding handler arm body.
- [x] Handler final binders warn when unused; intentionally ignored names such as `_value` suppress the warning.
- [x] The implementation separates reusable checked-AST value-use collection from unused-warning policy.
- [x] References record whether they originate from a local binder definition body.
- [x] The implementation uses resolved symbol identity and does not introduce a lexical scope tree for this warning.
- [x] Used bindings, shadowed bindings, and intentionally ignored named binders such as `_x` do not produce false positives.
- [x] Typecheck failures do not produce extra unused-local-value warnings in v1.
- [x] Warnings do not change success exit status, artifact generation, or execution when no errors are present.
- [x] CLI rendering, LSP diagnostics, and relevant fixtures distinguish warnings from errors.
- [x] Diagnostics use code `W2001`, message `unused local value binding`, and the agreed label/help wording.
- [x] Warning output order is stable and follows binder source locations.
- [x] Fixtures explicitly assert warning diagnostics instead of only checking success or error output.
- [x] Blackbox or whitebox tests lock the warning behavior.
